# VoC → 에이전트 자동 개선 파이프라인 — 설계

작성일: 2026-06-22 · 1차 대상 프로젝트: 블로그 챌린지

## 1. 목적
사용자가 만든 서비스의 VoC(불편/요청)를 수집하고, 에이전트들이 **파악 → 담당 배정 → 개선안 구상 → (승인) → 해결될 때까지 구현 → 완료보고**까지 자동화한다. 사람은 **승인**과 **커밋** 두 지점에만 개입한다.

## 2. 실행 모델 (확정)
- **Claude Code 세션이 오케스트레이터.** 별도 호스팅·스케줄러·상태머신 없음.
- 에이전트는 `.claude/agents/*.md` 정의. 메인 세션이 Agent/Task 도구로 팬아웃·검수.
- 트리거: 사용자가 "VoC 처리"를 지시(또는 텔레그램 메시지)하면 세션이 VoC 시트를 읽고 시작.

## 3. 에이전트 구성

| 에이전트 | 위치 | 역할 | 품는 지식 |
|---|---|---|---|
| **voc-router** (개선) | `~/.claude/agents/` (글로벌) | 전 프로젝트 VoC read·중복제거·분류·심각도 산정·**담당 PO 배정** | 프로젝트↔PO 매핑 레지스트리 |
| **po-blog-challenge** (PO) | `블로그-챌린지/.claude/agents/` | VoC 파악 → 스펙확인·브레인스토밍 → **PRD·납품조건** → 업무분배 → 검수 → **사인오프**. 통과까지 재지시 | 코어가치, `docs/spec.md`, `design-system/MASTER.md`(BI) |
| **dev-blog-challenge** | 〃 | 바닐라 JS(ES모듈)/GAS/Sheets TDD 구현 | 스택·코어스펙·`src/lib` 테스트 규칙 |
| **design-blog-challenge** | 〃 | 기획/디자인, MASTER 토큰만 사용, 안티클리셰 | 디자인시스템·UI 규칙 |
| **qa-blog-challenge** | 〃 | `npm test` + Playwright 렌더·클릭 검증 + 글로벌 `qa-*` 위임 | QA·회귀 체크리스트(메모리 규칙) |

- **확장 규칙**: 새 프로젝트 = `po-<프로젝트>` + 하위 3종을 그 프로젝트 `.claude/agents/`에 추가 + voc-router 레지스트리에 한 줄 등록.
- 루트 `my-claude-project/.claude/agents/qa-*`는 공용 폴백으로 유지. `qa-blog-challenge`가 필요 시 위임.

### PO 책임 정의
PO는 "문제·버그 없이 핵심가치 전달"을 목표로 한다. 핵심가치·스펙·BI는 불변 기준으로 품고, 모든 산출물이 이를 침해하지 않는지 검수한다. 책임감 있게 **납품기준 통과까지 하위 에이전트에 재지시**한다.

## 4. VoC 저장 (구글시트 새 탭 `VoC`)
컬럼: `id │ ts │ project │ channel │ phone? │ message │ status │ assignee │ resolution │ commit`
- `status`: `new → assigned → in_progress → resolved → committed`
- 앱(블로그 챌린지)에 **피드백 버튼** → GAS `submitVoc(project, message, phone?)` → 시트 append (`status=new`).
- voc-router는 GAS `getVoc(status=new)`로 read (CSV 폴백 가능).
- 비개발자·운영자도 시트에 직접 행 추가 가능.

## 5. 텔레그램
- **양방향**: 아웃바운드 알림 + 폰에서 승인. 컴퓨터(터미널) 승인도 병행 — 먼저 오는 응답 채택.
- 봇 토큰·chat_id는 **로컬 untracked 파일**(예: `~/.claude/.telegram`)에 보관. git·시트·코드에 값 금지.
- 세션이 Bot API를 curl로 호출. 승인 대기 = `getUpdates` 폴링(폰 '승인/반려') 또는 터미널 입력.

## 6. 흐름 (요청 7단계 매핑)
1. **수집**: 앱 피드백 버튼 → 시트 `VoC` 탭
2. **파악·배정**: voc-router read → 분류·중복제거 → 담당 PO 지정 (`status=assigned`)
3. **개선안**: PO가 PRD·납품조건 구상
4. **승인요청 푸시** ⛔게이트: 텔레그램으로 `VoC 원문 + 개선안` 전송
5. **승인**: 폰(텔레그램 버튼) 또는 터미널
6. **실행 루프**: PO가 dev/design/qa 팬아웃 → **PO 사인오프까지 반복** (`in_progress`)
7. **완료보고 푸시**: 텔레그램으로 변경요약·검증결과 → **사용자가 커밋**(전역 git 규칙) → `committed`

## 7. 납품기준 (Definition of Done)
- **단일 게이트 = PO 최종 사인오프.** PO는 원래 VoC가 실제 해결됐는지 대조 확인.
- PO는 사인오프 근거로 `npm test` 그린, 변경 성격에 맞는 `qa-*` PASS, UI 변경 시 Playwright 렌더·클릭 검증을 **요구·확인**한다. 기계적 체크리스트가 아니라 PO 책임 판단으로 종료.

## 8. 승인·커밋 게이트 (사람 개입 2지점)
- **승인(4단계)**: 개선안 실행 전. 반려 시 PO가 개선안 재구상.
- **커밋(7단계)**: 완료 후. 세션이 변경/커밋 메시지 준비, 사용자가 push 승인(전역 git 규칙 — add -A 금지, 코드/설정만 선별).

## 9. 범위 (1차 구현)
- 만든다: 시트 `VoC` 탭 + GAS `submitVoc`/`getVoc` + 앱 피드백 버튼 + 에이전트 5종 전부 + 텔레그램 알림/승인 유틸.
- 안 만든다: 다중 프로젝트 PO(블로그 챌린지만), 자동 스케줄 트리거(수동 지시), 상태머신 서비스.

## 10. 컴포넌트 경계 (독립 단위)
- **수집부**(시트탭+GAS+피드백버튼): 입력 = 사용자 피드백, 출력 = `VoC` 행. 파이프라인과 디커플.
- **voc-router**: 입력 = `VoC(new)`, 출력 = PO 배정된 구조화 항목. 프로젝트 무지(레지스트리만 앎).
- **PO**: 입력 = 배정 VoC, 출력 = PRD + 사인오프. 하위 에이전트 구현 디테일 무지(인터페이스로 지시).
- **하위(dev/design/qa)**: 입력 = PO 작업지시, 출력 = 산출물 + 검증결과.
- **텔레그램 유틸**: 입력 = 메시지/승인요청, 출력 = 전송·승인결과. 도메인 무지.
